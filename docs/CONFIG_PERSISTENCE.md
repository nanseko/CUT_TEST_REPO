# 설정 자동 저장 + 체크포인트별 하이퍼파라미터 로그/복원

## 1. "서버를 재시작할 때마다 경로가 초기화됨" — 원인과 수정

### 원인
`gui_config.json`의 기본 경로가 **상대경로**(`./gui_config.json`)로 되어 있었습니다.
Python의 상대경로는 **파일이 어디 있는지가 아니라 프로세스를 실행한 현재 작업 디렉토리(CWD)** 기준으로
풀립니다. 그래서 `gui.py`를 실행하는 방식(바탕화면 바로가기, 배치파일, 다른 터미널 위치 등)이
매번 조금씩 다르면 `./gui_config.json`이 **매번 다른 실제 파일**을 가리키게 되어, 이전에 저장한 설정을
못 찾고 기본값(빈 경로 등)으로 되돌아간 것처럼 보였습니다.

### 수정
```python
DEFAULT_CONFIG_PATH = os.path.join(REPO_ROOT, 'gui_config.json')   # REPO_ROOT = gui.py 파일 자신의 위치
```
`gui.py` 파일 자체의 위치에 고정된 **절대경로**로 바꿨습니다. 이제 **어디서 실행하든 항상 같은
`gui_config.json`** 을 읽고 씁니다. (검증: 완전히 다른 작업 디렉토리에서 실행해도 동일 경로로 resolve됨)

### + 자동 저장
기존엔 각 탭의 **"💾 저장" 버튼을 눌러야만** 값이 파일에 반영됐습니다(안 누르면 유실).
이제 **모든 입력값이 바뀔 때마다 자동으로 전체 설정이 저장**됩니다(각 위젯의 `change` 이벤트가
`do_save`를 실행). 화면 상단의 "자동 저장 상태" 표시줄에서 언제 저장됐는지 확인할 수 있고,
"저장" 버튼들은 여전히 남아있어 명시적으로 저장 시점을 확인하고 싶을 때 쓸 수 있습니다.

## 2. 체크포인트별 하이퍼파라미터 로그 + 불러오기

### 문제
`gui_config.json`은 **"GUI에서 마지막으로 쓴 설정" 딱 하나만** 기억합니다. 여러 실험(체크포인트)을
오가며 학습하면(예: `sar_resnet_v1` 학습 후 `sar_hrnet_v2`로 바꿔 학습), 예전 실험으로 돌아가
"이어서 학습"을 누르는 순간 **그 실험이 어떤 netG/attention/hrnet 설정으로 학습됐는지**가 이미
`gui_config.json`엔 최신 것(다른 실험 것)으로 덮어써져 있어 알 수 없습니다. 이 상태로 이어서 학습하면
**체크포인트 가중치와 현재 모델 구조가 어긋나 로드가 실패하거나 잘못될 수 있습니다.**

### 해결: 체크포인트마다 자체 설정 로그
학습을 시작할 때마다(신규든 이어서든) **그 실험의 전체 하이퍼파라미터 스냅샷**이
```
checkpoints_dir/<name>/gui_train_config.json
```
에 저장됩니다. `gui_config.json`(GUI 전체의 "마지막 상태")과 달리, 이건 **각 체크포인트 폴더에 귀속된
그 실험만의 기록**이라 다른 실험을 아무리 학습해도 덮어써지지 않습니다.

> 참고: CUT 원본도 `checkpoints_dir/<name>/train_opt.txt`(사람이 읽는 텍스트)를 남기지만, 이건
> GUI 위젯에 그대로 되불러오기 어려운 형식입니다. `gui_train_config.json`은 GUI가 바로 읽어
> 모든 탭 위젯에 되돌려놓을 수 있는 **전용 JSON**입니다.

### 탭 6: 체크포인트 설정 불러오기
1. **체크포인트(실험) 선택** 드롭다운에서 되돌아가고 싶은 실험을 고릅니다
   (`checkpoints_dir`를 스캔해 자동으로 채워짐, 🔄 새로고침 가능).
2. **📂 이 설정 불러오기** 클릭 → 탭 1~5의 모든 위젯(netG, attention, lambda_*, hrnet_* 등)이
   그 실험을 학습할 때 썼던 값으로 **한 번에 복원**됩니다.
3. 이제 탭 6에서 **"이어서 학습"을 체크**하고 학습을 시작하면, 정확히 그 설정으로 이어집니다.

**`continue_train` 체크박스 자체는 복원 대상에서 제외됩니다** — 이건 "이번에 이어서 할지"를 정하는
액션 버튼이지 실험의 고유 설정이 아니라서, 그대로 복원하면 처음 학습 스냅샷(대개 미체크 상태)으로
덮어써져 오히려 헷갈립니다. 사용자가 직접 체크하도록 남겨둡니다.

### 안전장치: 아키텍처 불일치 자동 경고
**"이어서 학습"을 체크하고 시작할 때마다**, 저장된 체크포인트 설정과 현재 화면의 설정을
자동으로 비교합니다 (`netG`, `normG`, `attention_type/reduction/encoder/resblocks/decoder`,
`no_antialias(_up)`, `hrnet_branches/modules/blocks` — 가중치의 실제 shape을 결정하는 항목들).

- **다르면**: 학습 시작 로그에 ⚠️⚠️ 경고와 함께 **정확히 어떤 키가 어떻게 다른지** 나열됩니다
  (막지는 않습니다 — 의도적인 아키텍처 변경일 수도 있으므로).
- **같으면**: ✅ 일치 확인 메시지가 남습니다.
- 이전 설정 로그가 아예 없는(이 기능 이전에 학습된) 체크포인트는 비교를 건너뜁니다.

### 예전(이 기능 이전) 체크포인트는?
`gui_train_config.json`이 없는 체크포인트도 드롭다운에 **"(설정 로그 없음)"** 표시로 나타나
(가중치 파일은 있으니) 최소한 존재는 확인할 수 있습니다. 불러오기를 누르면 "설정 로그가 없다"는
안내와 함께 수동 설정을 안내합니다 — 이후 그 실험을 다시 학습하면 그때부터는 로그가 남습니다.

## 3. 탭마다 데이터 폴더 지정 (전처리/학습/후처리/평가)

기존엔 `dataroot`를 **탭 1(데이터셋)에서만** 지정할 수 있었고, 후처리(탭 9)·평가(탭 8)는 항상
`results_dir/name/test_<epoch>/images/...` 라는 **고정된 규칙으로 자동 유도된 폴더**만 볼 수 있었습니다.
이제 각 탭에서 직접 폴더를 지정할 수 있고, **전부 자동저장 대상(CONFIG_KEYS)** 이라 서버를 껐다 켜도 남습니다.

| 탭 | 필드 | 기본 동작 |
|---|---|---|
| 2. 전처리 | 입력/출력 폴더 (`pp_in`/`pp_out`, 이미 있었음) | 상대경로 버그 수정 + 자동저장 추가 (`PP_CONFIG_PATH`도 절대경로로) |
| 3. 학습 | `dataroot` 미러 필드 | 탭 1과 같은 값을 이 탭에서도 바로 편집 가능(탭1의 `dataroot`에 반영·저장됨) |
| 8. 평가 | `eval_fake_dir`/`eval_real_a_dir`(신규), `eval_eo_dir`/`eval_real_b_dir`(기존을 CONFIG_KEYS로 승격) | 비워두면 기존처럼 `results_dir/name/test_<epoch>/...` 자동 사용, 채우면 **그 폴더로 평가** |
| 9. 후처리 | `rectify_input_dir`(신규) | 비워두면 자동(`fake_B`), 채우면 **그 폴더 전체**를 분석 |

**주의**: 탭 3의 `dataroot` 미러는 **탭3→탭1 방향으로만** 자동 반영됩니다(탭 1에서 바꾼 값이 탭 3
표시에 즉시 반영되려면 페이지 새로고침이 필요). 실제 저장되는 값은 항상 같은 `dataroot` 하나이므로
데이터 불일치 위험은 없습니다.

## 검증
`python tests/test_config_persistence.py`:
- 설정 경로가 절대경로이고 실행 위치(CWD)와 무관하게 항상 동일함(다른 프로세스/디렉토리로 실측)
- `build_ui()` 성공 + `CONFIG_KEYS` ↔ 위젯 완전 일치
- 자동 저장 라운드트립(저장→재로드)
- 체크포인트 설정 스냅샷 저장/로드, 레거시(로그 없는) 체크포인트 목록 표시
- 아키텍처 불일치 탐지(정확히 어떤 키인지) / 일치 시 무경고 / 비교 대상 없을 때 무경고
- 설정 불러오기 시 `continue_train`은 보존되고 나머지는 복원됨, 없는 실험 선택 시 안전 처리
- **실제 학습으로 엔드투엔드**: 신규 시작(스냅샷 생성) → 다른 아키텍처로 이어서 시도(경고 발생) →
  같은 아키텍처로 이어서 시도(일치 확인) 전부 실제 `train.py` 서브프로세스로 확인
- `PP_CONFIG_PATH`(전처리 탭) 절대경로 확인, 탭 8/9 폴더 필드가 `CONFIG_KEYS`에 포함됨을 확인
- 탭 8/9 폴더 override가 실제로 임의 폴더를 읽는지는 `python tests/test_rectify.py`
  (`gui.cut_rectify` 부분)와 `evaluation/evaluate.py::run_evaluation`의 `fake_dir`/`real_a_dir`
  파라미터 경로로 확인
