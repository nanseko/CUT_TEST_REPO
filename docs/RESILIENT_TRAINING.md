# 장시간(1~2일) 학습이 도중에 멈추는 문제 — 원인과 대책

**증상**: 학습이 에러로 죽는 게 아니라 **진행이 멈춰버림**(GPU/로그가 더 안 올라감). 수동으로 중단 후 다시 시작하면 정상 동작.

## 원인 진단

먼저 확인할 사실: **Web-UI(브라우저) 연결과 실제 학습은 이미 분리되어 있습니다.** `▶ 학습 시작`을 누르면 학습은 gui.py 서버 프로세스의 **백그라운드 스레드 + 서브프로세스**로 돌아가고, 브라우저 탭/네트워크 연결은 단지 진행 상황을 "보여주는" 창일 뿐입니다. 즉 **브라우저를 닫거나 네트워크가 끊겨도 gui.py 프로세스 자체가 살아있는 한 학습은 계속됩니다.**

그래서 "정지되는 것은 아닌데 멈추는" 증상은 보통 아래 중 하나입니다:

1. **PC 절전/최대절전 모드** — Windows는 기본적으로 일정 시간 후 절전 모드로 들어갈 수 있습니다(앱이 돌고 있어도!). **가장 흔한 원인입니다.**
2. **Windows 자동 업데이트로 인한 재부팅** — 활성 시간이 설정 안 돼 있으면 야간에 재부팅되며 학습 프로세스 전체가 사라집니다.
3. **DataLoader 워커 정지** — `num_threads > 0`(멀티프로세스 데이터 로딩)이 장시간 실행 중 멈추는 경우가 있습니다. 특히 Windows는 Linux보다 이 문제에 취약합니다(spawn 방식 멀티프로세싱).
4. **네트워크 드라이브 I/O 정지** — `dataroot`/`checkpoints_dir`가 NAS/공유폴더(네트워크 경로)에 있으면, 네트워크가 잠깐 끊겨도 OS의 파일 읽기/쓰기 호출이 **에러 없이 무한정 블로킹**될 수 있습니다. "네트워크가 끊겨도 계속 동작해야" 라는 우려와 정확히 맞아떨어지는 원인입니다.
5. GPU/드라이버 레벨 행(hang), 디스크 I/O 정지(백신 실시간 검사 등).

## 소프트웨어 대책: 자동 행(hang) 감지 + 재시작 (이미 적용됨)

**탭 6. 학습 실행/모니터링**에 워치독을 추가했습니다:

- **행(hang) 감지 시간(분)** 동안 학습 로그에 진행(iters/epoch)이 전혀 없으면, 프로세스를 **자동으로 강제 종료**하고 **마지막 체크포인트부터 `--continue_train` 로 자동 재시작**합니다.
- **최대 자동 재시작 횟수**까지 반복하며, 초과하면 명확한 오류 메시지를 남기고 멈춥니다(무한 재시도 방지).
- 이유 없이 죽는 경우(비정상 종료 코드)도 동일하게 자동 재시작됩니다.
- 재시작마다 로그에 시각·사유·시도 횟수가 남아 나중에 감사(audit) 가능합니다.
- 원인 4번(네트워크 드라이브 I/O 정지)은 워치독이 **완화**는 하지만(멈춰있다 감지되면 강제 kill 후 재시작), 애초에 로컬 드라이브를 쓰는 게 근본 해결책입니다(아래 참고).

기본값: 20분 무진행 → 재시작, 최대 20회. 데이터셋이 커서 epoch이 원래 20분보다 오래 걸리면 시간을 넉넉히 늘리세요(오탐 방지).

## OS 레벨 대책 (코드로는 해결 불가 — 반드시 설정 필요)

**① Windows 절전 모드 끄기** (1번 원인의 근본 해결):
- 설정 → 시스템 → 전원 및 절전 → **"화면 끄기"/"PC 절전 모드"를 모두 "안 함"** 으로 설정
- 또는 명령 프롬프트(관리자)에서:
  ```
  powercfg /change standby-timeout-ac 0
  powercfg /change monitor-timeout-ac 0
  ```
- 노트북이면 **뚜껑을 닫아도 절전 안 함** 설정도 확인(전원 옵션 → 덮개 닫을 때의 동작).

**② Windows Update 자동 재부팅 방지** (2번 원인):
- 설정 → Windows Update → 고급 옵션 → **활성 시간**을 학습 도는 시간대 전체로 설정(최대 18시간 범위 제한이 있으니, 정말 장시간이면 그룹 정책/재부팅 알림 설정도 함께 확인).

**③ 데이터/체크포인트는 로컬 드라이브에** (4번 원인의 근본 해결):
- `dataroot`, `checkpoints_dir` 를 **네트워크 드라이브(NAS, 공유폴더, 매핑된 드라이브)가 아니라 로컬 SSD/HDD 경로**로 두세요. 네트워크 스토리지를 꼭 써야 한다면, 학습 시작 전에 데이터를 로컬로 복사하고 체크포인트도 로컬에 썼다가 끝난 뒤 옮기는 방식을 권장합니다.

**④ `num_threads` 를 낮게** (3번 원인 완화):
- Windows에서 장시간 학습 시 `num_threads=0`(메인 프로세스에서 로딩) 또는 낮은 값(2 이하)이 멀티프로세스 워커 정지 위험을 줄입니다. 탭 3의 `num_threads` 에서 조절하세요.

**⑤ 백신 실시간 검사 예외 설정**:
- `checkpoints_dir`, `dataroot`, 저장소 폴더를 Windows Defender(또는 사용 중인 백신)의 **실시간 검사 제외 목록**에 추가하면 대용량 파일 반복 접근으로 인한 I/O 지연을 줄일 수 있습니다.

## 요약 체크리스트

- [ ] 절전/최대절전 모드 끔 (①)
- [ ] Windows Update 활성 시간 설정 (②)
- [ ] dataroot/checkpoints_dir 가 로컬 드라이브인지 확인 (③)
- [ ] num_threads 를 낮게(0~2) (④)
- [ ] 백신 예외 등록 (⑤)
- [ ] 탭 6의 행 감지 시간/최대 재시작 횟수 확인 (기본값 20분/20회면 충분히 안전)

이 다섯 가지를 다 해두면, 소프트웨어 워치독과 결합해 **PC만 켜져 있으면(그리고 gui.py 프로세스가 살아있으면) 1~2일 무인 학습이 안정적으로 완료**됩니다.

## 검증

`python tests/test_training_watchdog.py` — 가짜 학습 스크립트로 (1) hang 감지 후 자동 재시작, (2) 비정상 종료 후 자동 재시작, (3) 최대 재시작 횟수 초과 시 정지, (4) 재시작 대기 중 "중단" 요청 시 즉시 취소를 확인합니다(실제 다중 학습 없이 수 초 내 검증).
