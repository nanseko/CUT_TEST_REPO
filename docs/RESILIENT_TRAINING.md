# 장시간 작업이 도중에 멈추는 문제 (행/hang) — 원인과 대책

**증상**: 학습·추론·하이퍼파라미터 자동 탐색 도중 에러로 죽는 게 아니라 **진행이 멈춰버림**(GPU/로그가
더 안 올라감). 수동으로 중단 후 다시 시작하면 정상 동작. **학습에서만 생기는 게 아니라, train.py/test.py를
서브프로세스로 실행하는 모든 작업**(추론, 하이퍼파라미터 탐색의 각 trial)에서 같은 근본 원인으로 발생할
수 있습니다 — 아래 워치독은 그래서 한 곳이 아니라 **시스템 전체에 동일하게 적용**되어 있습니다.

## 원인 진단

먼저 확인할 사실: **Web-UI(브라우저) 연결과 실제 학습은 이미 분리되어 있습니다.** `▶ 학습 시작`을 누르면 학습은 gui.py 서버 프로세스의 **백그라운드 스레드 + 서브프로세스**로 돌아가고, 브라우저 탭/네트워크 연결은 단지 진행 상황을 "보여주는" 창일 뿐입니다. 즉 **브라우저를 닫거나 네트워크가 끊겨도 gui.py 프로세스 자체가 살아있는 한 학습은 계속됩니다.**

그래서 "정지되는 것은 아닌데 멈추는" 증상은 보통 아래 중 하나입니다:

1. **PC 절전/최대절전 모드** — Windows는 기본적으로 일정 시간 후 절전 모드로 들어갈 수 있습니다(앱이 돌고 있어도!). **가장 흔한 원인입니다.**
2. **Windows 자동 업데이트로 인한 재부팅** — 활성 시간이 설정 안 돼 있으면 야간에 재부팅되며 학습 프로세스 전체가 사라집니다.
3. **DataLoader 워커 정지** — `num_threads > 0`(멀티프로세스 데이터 로딩)이 장시간 실행 중 멈추는 경우가 있습니다. 특히 Windows는 Linux보다 이 문제에 취약합니다(spawn 방식 멀티프로세싱).
4. **네트워크 드라이브 I/O 정지** — `dataroot`/`checkpoints_dir`가 NAS/공유폴더(네트워크 경로)에 있으면, 네트워크가 잠깐 끊겨도 OS의 파일 읽기/쓰기 호출이 **에러 없이 무한정 블로킹**될 수 있습니다. "네트워크가 끊겨도 계속 동작해야" 라는 우려와 정확히 맞아떨어지는 원인입니다.
5. GPU/드라이버 레벨 행(hang), 디스크 I/O 정지(백신 실시간 검사 등).

## 소프트웨어 대책: 자동 행(hang) 감지 (시스템 전체 적용됨)

**처음엔 탭 6(학습)에만 워치독이 있었는데, 하이퍼파라미터 자동 탐색(탭 10)이나 추론(탭 7)에서도 같은 종류의
행 현상이 보고되어 — 실제로 확인해보니 두 곳 다 워치독이 전혀 없었습니다(순수 블로킹 대기라 한 번 멈추면
영원히 멈춤). 그래서 워치독 로직을 `util/subprocess_watchdog.py` 하나로 통합해 train.py/test.py를 실행하는
모든 지점에 동일하게 적용**했습니다.

| 탭 | 대상 | 멈추면? |
|---|---|---|
| 6. 학습 실행/모니터링 | `train.py` | 강제 종료 후 **`--continue_train`로 자동 재시작**(최대 재시작 횟수까지) |
| 7. 추론/테스트 | `test.py` | 강제 종료 후 **오류 메시지 표시**(재시작은 안 함 — 다시 버튼을 누르면 재시도) |
| 10. 하이퍼파라미터 자동 탐색 | 각 trial의 `train.py`/`test.py` | 강제 종료 후 **그 trial만 실패 처리, 다음 trial로 계속 진행** — 탐색 전체가 멈추지 않음 |

공통 동작:
- **행(hang) 감지 시간(분)** 동안 진행(iters/epoch) 로그가 전혀 없으면 프로세스를 **자동으로 강제 종료**합니다.
- 이유 없이 죽는 경우(비정상 종료 코드)도 "멈춤"과 동일하게 처리됩니다.
- 로그에 시각·사유가 남아 나중에 감사(audit) 가능합니다.
- 원인 4번(네트워크 드라이브 I/O 정지)은 워치독이 **완화**는 하지만(멈춰있다 감지되면 강제 kill), 애초에
  로컬 드라이브를 쓰는 게 근본 해결책입니다(아래 참고).

기본값: 20분 무진행 → 감지. 데이터셋이 커서 epoch이 원래 20분보다 오래 걸리면 시간을 넉넉히 늘리세요
(오탐 방지). 각 탭에서 개별적으로 조절 가능합니다(탭 6/7/10에 각각 "행(hang) 감지 시간" 필드).

**재시작 vs 실패 처리, 왜 다른가**: 학습(탭 6)은 체크포인트가 있어 `--continue_train`으로 손실 없이 재개할
수 있지만, 추론(탭 7)의 `test.py`는 애초에 "이어서"라는 개념이 없는 1회성 작업이라 자동 재시작 대신 실패로
끝내고 사용자가 다시 누르게 합니다. 하이퍼파라미터 탐색(탭 10)의 trial 하나는 애초에 여러 후보 중 하나일
뿐이라, 그 trial을 재시도하느라 시간을 쓰기보다 실패로 기록하고 나머지 trial로 넘어가는 게 전체 탐색
관점에서 더 낫습니다(다른 trial들이 이미 충분한 커버리지를 제공).

## OS 레벨 대책 (코드로는 해결 불가 — 반드시 설정 필요)

**① Windows 절전 모드 끄기** (1번 원인의 근본 해결):
- 설정 → 시스템 → 전원 및 절전 → **"화면 끄기"/"PC 절전 모드"를 모두 "안 함"** 으로 설정
- 또는 명령 프롬프트(관리자)에서:
  ```
  powercfg /change standby-timeout-ac 0
  powercfg /change monitor-timeout-ac 0
  ```
- 노트북이면 **뚜껑을 닫아도 절전 안 함** 설정도 확인(전원 옵션 → 덮개 닫을 때의 동작).

**② Windows Update 자동 재부팅 방지** (2번 원인):
- 설정 → Windows Update → 고급 옵션 → **활성 시간**을 학습 도는 시간대 전체로 설정(최대 18시간 범위 제한이 있으니, 정말 장시간이면 그룹 정책/재부팅 알림 설정도 함께 확인).

**③ 데이터/체크포인트는 로컬 드라이브에** (4번 원인의 근본 해결):
- `dataroot`, `checkpoints_dir` 를 **네트워크 드라이브(NAS, 공유폴더, 매핑된 드라이브)가 아니라 로컬 SSD/HDD 경로**로 두세요. 네트워크 스토리지를 꼭 써야 한다면, 학습 시작 전에 데이터를 로컬로 복사하고 체크포인트도 로컬에 썼다가 끝난 뒤 옮기는 방식을 권장합니다.

**④ `num_threads` 를 낮게** (3번 원인 완화):
- Windows에서 장시간 학습 시 `num_threads=0`(메인 프로세스에서 로딩) 또는 낮은 값(2 이하)이 멀티프로세스 워커 정지 위험을 줄입니다. 탭 3의 `num_threads` 에서 조절하세요.

**⑤ 백신 실시간 검사 예외 설정**:
- `checkpoints_dir`, `dataroot`, 저장소 폴더를 Windows Defender(또는 사용 중인 백신)의 **실시간 검사 제외 목록**에 추가하면 대용량 파일 반복 접근으로 인한 I/O 지연을 줄일 수 있습니다.

## 검증

- `python tests/test_subprocess_watchdog.py` — 공용 워치독 primitive(`util/subprocess_watchdog.py`) 자체:
  정상 완료/hang 감지/비정상 종료/즉시 중단/제너레이터 인터페이스/진행으로 안 치는 라인 필터링, 그리고
  탭 7(추론)이 실제로 이 워치독을 쓰는지까지 확인.
- `python tests/test_training_watchdog.py` — 탭 6(학습)의 통합: (1) hang 감지 후 자동 재시작, (2) 비정상
  종료 후 자동 재시작, (3) 최대 재시작 횟수 초과 시 정지, (4) 재시작 대기 중 "중단" 요청 시 즉시 취소.
- `python tests/test_hparam_search.py` — 탭 10의 통합: **실제로 한 trial을 hang시켜서 전체 탐색이 멈추지
  않고, 그 trial만 실패로 기록된 뒤 나머지 trial로 계속 진행되어 탐색이 정상 완료되는지** 실측 확인
  (`test_stalled_trial_does_not_block_the_whole_search`).
